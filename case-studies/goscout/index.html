GoScout: Building a Full-Auto Sports Analytics Product from Zero

The problem

Sports betting analysis in Brazil runs on tipsters who read stats by hand and post picks with no track record anyone can audit. I built GoScout to replace that with a real data pipeline. It ingests live football data across dozens of leagues, calibrates its own confidence thresholds against historical results, and publishes betting entries automatically, with real bookmaker odds attached so every pick can be checked against what the market was offering.

It's a subscription product with Hubla-based billing, running end to end without a human in the publishing loop. Ingestion, scoring, gating, publishing, live tracking, and settlement all run on their own.

Architecture

Ingestion Stats engine Calibration Publishing gates Payments Settlement

Data pipeline: a cron-driven engine on a VPS pulls fixtures, team and player stats, lineups, and odds from a sports-data REST API across 55 active leagues, inside a hard budget of 75,000 requests per day that I manage in code. Every job runs against a per-run request cap: going over quota shuts down the whole pipeline for the rest of the day. Live matches get a dedicated ingest-live job on a 1-minute cadence, separate from the slower daily ingestion jobs, so live odds and stats never compete with historical backfill for the same budget.

Stats engine: markets like player shots, corners, cards, and match goals get calibrated against real historical results before they're allowed to publish. A market goes live only after backtesting shows it clears a validation threshold on held-out data. Every active market carries a documented blind-training pass behind it.

Publishing gates: a picked market has to clear a value gate against the real bookmaker odd, not just the model's own confidence, before it can publish, plus a minimum combined-odds floor. I enforced one principle across every gate: not publishing beats publishing something wrong or weak. Several markets stay switched off in production today because they never cleared validation, even when the model favored using them.

Payments & account activation: subscription billing runs through Hubla via webhook. subscription.activated and subscription.deactivated events upsert account status by email. This is where the most instructive production incident happened, covered below.

Settlement: a separate cron job checks every published entry against final match stats and closes it out as hit, miss, or void automatically, including partial-leg tracking while a match is still live.

Production incident: the payment webhook that never made a sound

This is the piece I'd point to first. It's less about writing code and more about the debugging a GTM/RevOps role needs. Subscribers were paying and not getting access. The obvious suspect, an auth token bug, had already been found and fixed once. Then it happened again, to different subscribers.

1 · Symptom

Two paying subscribers reported no access after checkout. It looked like a repeat of the already-fixed token bug. It wasn't.

2 · Investigation

I checked the webhook audit table, which logs every inbound event before processing. It had zero rows since the integration existed. That ruled out a bug in the handler. The handler was never being reached.

3 · Root cause

The webhook URL registered in the payment provider's dashboard pointed at the bare apex domain. Vercel 308-redirects that domain to the www subdomain at the routing layer. The payment provider doesn't follow redirects on webhook calls, so every webhook call died before reaching application code. No error, no 401, no log line, because the request never arrived: a routing configuration issue, invisible from inside the app.

4 · Fix + verification

I corrected the URL in the provider dashboard to the canonical www host, then verified with a real simulated webhook POST: 200 response, event applied, row written to the audit table.

5 · Systemic fix, not just a patch

A wrong URL is the kind of thing that breaks silently again: the provider changes something, DNS changes, someone edits the dashboard. Instead of fixing just the one URL, I shipped a watchdog cron that watches for silence, not errors. If the audit table goes more than 24 hours without a new row, it fires a Telegram alert. That covers this failure mode and any future one that produces the same symptom, without requiring me to predict the next cause.

What this demonstrates

Root-cause debugging under production pressure

Re-reading the same code twice wouldn't have solved the webhook incident. It took forming a hypothesis, checking an audit trail, and identifying an infrastructure-layer cause, a routing redirect, instead of stopping at the first plausible app-layer explanation.

Building for the next failure, not just this one

The watchdog doesn't detect a wrong webhook URL. It detects a signal gone quiet, a more useful thing to monitor that covers failure modes I haven't seen yet.

Operating real API/data constraints

A hard daily request budget across dozens of concurrent data sources, enforced in code rather than assumed. Enrichment and scoring pipelines against rate-limited CRM and data-provider APIs need the same discipline.

Shipping gated automation, not just automation

Every market and every entry passes through explicit value and confidence gates before publishing. Lead scoring and signal-to-action workflows need the same pattern: they can't fire on noise.
